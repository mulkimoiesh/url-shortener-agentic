# 06 — Engineering Journal

This document is a narrative account of building and hardening this system — not a spec, but the actual process: what was genuinely difficult, what broke under real end-to-end testing against a live LLM, and what was learned fixing it. Where [04-Design-Decisions.md](04-Design-Decisions.md) states *what* was decided and *why* it should work, this document covers *what actually happened* when it was run for real, repeatedly, against a live provider — which is where most of the substantive engineering in this submission actually occurred.

## 1. Major Design Challenges

### 1.1 Workspace Isolation

Every run needed to operate on a fully independent, disposable copy of the repository — fast enough to survive several retries per run without becoming the bottleneck, and structurally incapable of ever touching the live product by accident. The design (copy-on-first-use into `run-artifacts/{runId}/workspace/`, `/apply` as the only path back) is simple to state but exposed two real problems once it was actually exercised:

- **Path-separator inconsistency.** The workspace's file index was built from `Path::toString`, which emits backslashes on Windows, while every architecture-proposed and plan-returned path in the system uses forward slashes. The two never matched, so the validator treated an existing file as a completely different, unrelated file — legitimate `MODIFY`s were rejected as "duplicate class" or "unplanned CREATE." Fixed by normalizing every indexed path to forward slashes at the point of enumeration.
- **A relative `repo-root` is only as reliable as the process's working directory.** `Path.of(repoRoot).normalize()` (rather than `.toAbsolutePath().normalize()`) meant a bare `..` never actually resolved against anything until it happened to be combined with a *longer* path later — so two different code paths derived from the same config value silently disagreed on where the workspace root actually was, and every file write failed a security check with "Refusing to write outside of..." even though nothing was actually wrong with the target path. This is covered in more depth in §3.

### 1.2 Brownfield Preservation

The hardest sustained problem in the whole system: how does an agent that only sees "these files, modify them" avoid silently breaking behavior it wasn't told about? This took several iterations to get right:

1. First pass: detect when a `MODIFY` plan drops an existing REST mapping or public method, by diffing old vs. new content. Necessary, but insufficient — it caught the input side (the *file being changed*) but not files the plan didn't mention at all.
2. Second pass: an "unplanned but the file already exists" case was being rejected outright, even when the unplanned touch was the *correct* thing to do (e.g. a DTO field addition Architecture's design simply forgot to flag as impacted). Reclassified as an implicit `MODIFY`, validated the same way a planned one is.
3. Third, and largest, pass: none of the above helps when the class being changed has a **public shape** (a constructor, a method signature) that an *existing test file* — which Implementation has zero visibility into — calls directly. This is not an edge case; it is the *normal* shape of a brownfield change (adding a field to a request record is about as ordinary as brownfield work gets), and it reproduced on every single attempt, regardless of which LLM sample came back. Closing it required a real, scoped feature addition — `TestImpactAnalyzer` — not another validator tweak. Full rationale in [04-Design-Decisions.md §8](04-Design-Decisions.md).

### 1.3 Retry Logic

Getting the retry loop to converge exactly at its stated bound, rather than approximately, turned out to matter more than expected — see §3 for the exact off-by-one that shipped first. The harder problem surfaced only once the system was run against a real, rate-limited provider: two workflow-level retries fired back-to-back (the ordinary case — a plan that needs one small correction) reliably land inside the *same* one-minute token-quota window, so both retry attempts died to throttling before either ever got a real chance to run, and the bounded retry budget was spent on a problem the retry loop was never meant to solve. This forced a real design split: workflow-level retry for "the plan was wrong," a separate backoff layer at the LLM client for "the provider is momentarily unavailable." Full rationale in [04-Design-Decisions.md §7](04-Design-Decisions.md).

### 1.4 Groq Integration

Wiring a second real provider behind the existing `AgentLlmClient` abstraction was mechanically simple — the abstraction did its job, zero agent code changed. The actual engineering challenge was operational, not architectural: running the four required scenarios end-to-end, repeatedly, against a free-tier account surfaced per-minute and per-day token quotas, and the discovery that several successive "different" API keys resolved to the *same* underlying organization and thus the *same* shared quota pool. None of this is a code defect — it's the ordinary reality of validating against a live model rather than a mock — but it directly shaped the retry/backoff work in §1.3 and meant every fix in §3 was verified against real, sometimes-inconvenient model behavior rather than a convenient canned response.

### 1.5 JSON Parsing

Every agent asks the model to "respond with ONLY valid JSON, no markdown fences, no commentary" — and real models comply most of the time, not all of the time. `JsonExtractionUtil` strips a leading/trailing code fence and extracts the outermost `{...}` span before parsing, and every agent wraps the parse in a try/catch that turns a malformed response into a clean `FAILURE` decision-log entry (which then flows into the ordinary retry path) rather than an uncaught exception. This is a small utility, but it is the reason a formatting quirk from the model degrades gracefully into "try again" instead of crashing the run.

## 2. Key Architectural Decisions

The complete decision record — including alternatives considered and rejected — lives in [04-Design-Decisions.md](04-Design-Decisions.md). In summary, the decisions that mattered most in practice:

- **A hand-rolled graph engine**, not a framework — so every governance mechanic (gate, retry, block) is inspectable in ~150 lines rather than hidden behind configuration.
- **`ChangePlanner` is authoritative over CREATE/MODIFY**, never the LLM — closes the single most common LLM failure mode (inventing a duplicate class) structurally.
- **Two independent retry layers**, never sharing a counter — see §1.3.
- **`TestImpactAnalyzer` bridges a real blind spot** rather than giving Implementation unrestricted test-file visibility — bounded prompt growth, and a validator check that a discovered test gets *fixed*, not deleted.
- **Docs and Release are templated, not LLM calls** — the data is already structured; asking a model to restate it introduces paraphrasing risk for no benefit, and the release checklist specifically needs to be trustworthy since a human signs off against it.
- **Fail fast on `repo-root` misconfiguration** at startup, rather than let it silently corrupt every run-artifacts path until a confusing failure surfaces minutes later (see §3).

## 3. Issues Resolved During Development

The following were all found by actually running the system end-to-end against a live LLM (not by static review), root-caused, fixed, and re-verified by re-running the affected scenario. Listed roughly in the order they surfaced.

| # | Symptom | Root Cause | Fix |
|---|---|---|---|
| 1 | `NullPointerException` inside `ImplementationValidator` whenever a class had no `extends` clause | `Set.of(...).contains(null)` throws — `stripGenerics(null)` returned `null` and was passed straight into an immutable set's `contains()` | Null-guard before the `contains()` call in `extendsSpringDataRepository` |
| 2 | Bounded retry occasionally allowed 3 attempts instead of 2 | Retry counter was incremented *before* checking the cap, so the exhausting failure overshot the bound | Check `retriesFor(...) >= MAX` before incrementing, not after |
| 3 | Legitimate `MODIFY`s rejected as duplicate/unplanned files | Workspace file index used `Path::toString`, emitting backslashes on Windows; every other path in the system uses forward slashes | Normalize indexed paths to `/` at enumeration time |
| 4 | A file that already used `@PostMapping` was flagged as "missing import" even when nothing about the import situation changed | The missing-import check ran unconditionally, ignoring whether the annotation was already present before this change | Only flag annotations *newly introduced* by the diff, not ones already present in the original file |
| 5 | An unrelated new utility class (`ShortCodeGenerator`) got redirected onto an existing, unrelated controller (`ShortenController`) | `ClassNameEquivalence`'s substring fallback matched on any shared token across multi-token names — "short" is trivially a substring of "shorten" | Restrict the substring fallback to names that reduce to exactly one core token each |
| 6 | A plan that preserved an endpoint was still rejected as "removed existing REST mapping" | The mapping-extraction regex only recognized the positional `@PostMapping("/x")` form, not the equally valid `@PostMapping(value = "/x")` form | Extended the regex to accept an optional `value=`/`path=` attribute |
| 7 | `gradlew.bat` "is not recognized" even though the file genuinely existed in the workspace | Windows' `CreateProcess` (what `ProcessBuilder` uses) cannot launch a `.bat` file directly — needs `cmd.exe` as the interpreter | Invoke via `cmd.exe /c <wrapper> ...` on Windows, with a fully-qualified path (not a bare filename relying on directory search) |
| 8 | `CodebaseContextService` hit the exact same "refusing to write outside root" failure as `WorkspaceService` had | Same root cause as issue #3's sibling bug: `.normalize()` called before `.toAbsolutePath()` on a relative `repo-root`, so the `..` never actually collapsed | Same fix: resolve to an absolute path before normalizing |
| 9 | A plan that genuinely needed to touch an existing (but architecturally unplanned) file was always rejected | `validateUnplannedFile` treated "unplanned + file already exists + marked CREATE" as an unconditional violation | Reclassify as an implicit `MODIFY` and validate it the same way a planned `MODIFY` is validated |
| 10 | A Java `record`'s own declaration line was mistaken for a "public method" named after the record | The public-method regex's non-greedy return-type wildcard swallowed the word `record` as if it were a type | Added a negative lookahead excluding `class`/`interface`/`enum`/`record` immediately after `public` |
| 11 | A legitimate corrective rewrite on retry 2 was rejected for "removing" methods that were never part of the real codebase | The "don't silently drop things" check diffed against the *workspace's current* content, which a previous (still-broken) retry attempt had already overwritten — not the true original | Added `readLiveProductFile()`; the validator now always diffs against the untouched live file, never a prior attempt's in-progress state |
| 12 | Audit log said "Design deferred pending clarification" even when a real, non-empty design had just been produced | The logging condition checked the *original scenario type* (`== AMBIGUOUS`) instead of whether *this specific* result was actually empty | Base the log message purely on `impactedFiles` emptiness |
| 13 | The ambiguous-scenario demo script reported success but the run was actually stuck one gate short of `COMPLETED` | A genuinely ambiguous request needs **two** separate `ARCHITECTURE` gate approvals (first triggers the re-plan, second approves its output); the script only had one conditional approval step | Replaced the single check with a bounded approval loop |
| 14 | Calling `/approve` or `/resolve-block` on a run in the wrong state returned a raw, unhelpful HTTP 500 | `WorkflowEngine`'s `IllegalStateException` was uncaught at the controller layer | Catch and re-throw as `ResponseStatusException(CONFLICT, ...)`, matching the pattern already used by `/apply` |
| 15 | The governance-demo's health-check feature reproducibly failed to compile | The LLM's idiomatic (correct) choice — Spring Boot Actuator's `HealthIndicator` — genuinely wasn't a dependency in `shortener-service`'s `build.gradle` | Added `spring-boot-starter-actuator` — the actual missing piece, not a prompt workaround |
| 16 | A brownfield change that added a field to an existing record broke a pre-existing integration test's positional constructor call, on every attempt, regardless of LLM sample | Implementation had zero visibility into `src/test/**` — a real architectural gap, not LLM variance (see §1.2) | Built `TestImpactAnalyzer`: discovers impacted tests by naming convention + content reference, includes them in the prompt with explicit "fix, don't weaken" instructions, and validates them like any other `MODIFY` |
| 17 | The application worked via `./gradlew bootRun` but failed with a confusing `gradlew.bat not recognized` error when launched from IntelliJ's green Run button | IntelliJ's default run configuration had no explicit working directory set, and resolved one level above the actual repo root — silently corrupting every `run-artifacts` path until the failure surfaced deep inside a `TESTING` stage | Pinned `WORKING_DIRECTORY` explicitly in both the local run configuration and a checked-in `.run/OrchestratorApplication.run.xml`; added a `@PostConstruct` fail-fast check in `WorkspaceService` so any future working-directory mismatch fails immediately at startup with an actionable message, instead of minutes later with a cryptic one |

Every fix above was verified by re-running the specific scenario that had exposed it, not by inspection alone — see [03-Workflow-Scenarios.md](03-Workflow-Scenarios.md) for the real, evidenced runs that resulted once these were resolved.
